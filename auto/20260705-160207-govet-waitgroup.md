# Govet WaitGroup Example

Commit: `2642ee6 Add govet waitgroup example`

## Context

The planned example was `govet/waitgroup_add_inside_goroutine`: `WaitGroup.Add` is called inside the goroutine it is supposed to track.

## Options

1. Skip the example because older Go versions did not always expose a `waitgroup` vet analyzer.
   Score: 3/10. Safe, but current Go 1.26.4 does include the analyzer.

2. Verify analyzer availability first, then implement it through `golangci-lint`.
   Score: 10/10. Confirms the toolchain behavior before adding the example.

3. Use direct `go vet -waitgroup`.
   Score: 5/10. Useful for investigation, but the repo plan says `govet` examples should go through `golangci-lint`.

4. Make the bad `SendAll` fail at runtime.
   Score: 4/10. The static analyzer would no longer be the main teaching signal.

5. Make the normal test wait for eventual side effects, while `govet` reports the broken wait contract.
   Score: 8/10. The example stays runnable and demonstrates why static analysis matters even when a test passes.

## Chosen

Options 2 and 5.

I confirmed `go tool vet help` lists `waitgroup`, then implemented a mailer where `SendAll` calls `wg.Add(1)` inside each worker goroutine.

## Why

The analyzer is present in the active Go 1.26.4 toolchain and `golangci-lint` v2.12.2 forwards it correctly. The example shows a real lifecycle ordering bug: `Wait` can observe a zero counter and return before workers register themselves.
