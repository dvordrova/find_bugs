# Govet Copylocks Example

Commit: `a72eaf5 Add govet copylocks example`

## Context

The planned example was `govet/copylocks`: `govet` through `golangci-lint` reports copying a type that contains a lock.

## Options

1. Run `go vet` directly.
   Score: 5/10. Simpler, but the plan explicitly said `go vet` examples should go through `golangci-lint`.

2. Use plain `golangci-lint` as a Go tool dependency, with `.golangci.yaml` enabling only `govet`.
   Score: 9/10. Matches production CI usage and keeps the example focused.

3. Reuse the NilAway custom `golangci-lint` build.
   Score: 3/10. Unnecessary and confusing because `govet` is built into normal `golangci-lint`.

4. Demonstrate copied lock via a function argument.
   Score: 7/10. Valid, but a value receiver on a service type is a clearer API-shape bug.

5. Demonstrate copied lock via a method value or assignment.
   Score: 6/10. Interesting but less direct for a first `govet` example.

## Chosen

Options 2 and 4's simpler sibling: a value receiver method on `AccountLedger`.

## Why

The value receiver makes the copied-lock problem visible at the API boundary. The normal program and tests pass, while `golangci-lint`/`govet` flags the unsafe receiver.
