# Как внести изменения

Этот репозиторий - каталог маленьких запускаемых Go-примеров багов и инструментов, которые умеют их ловить.

Держи изменения сфокусированными: один пример должен объяснять один паттерн бага или одно поведение инструмента.

## Добавить новый пример

1. Создай самодостаточную директорию примера:

   ```text
   <tool_or_problem>/<category>/
   ```

2. Обычно внутри должны быть:

   ```text
   go.mod
   Makefile
   main.go
   README.md
   README.ru.md
   ```

3. Пользовательские Make targets должны быть очевидными:

   ```sh
   make run
   make lint
   make test
   ```

   Используй tool-specific target только если `lint` плохо подходит.

4. Maintainer и CI targets размещай ниже пользовательских.

   `ci-test` должен перегенерировать committed snapshot logs, например `lint.logs` или `lint-fixed.logs`.

5. Фиксируй версии инструментов в Makefile примера.

   Не используй `go get -u` для обычных обновлений.

6. Перегенерируй snapshots:

   ```sh
   make test-update
   ```

7. Проверь изменившиеся `*.logs`.

   Snapshot diff должен быть понятен человеку, который изучает инструмент.

8. Запусти проверку репозитория:

   ```sh
   make test
   ```

## Документация

README каждого примера должен объяснять:

- какой баг или поведение показаны;
- как это запустить;
- какой вывод ожидается;
- как читать отчет инструмента;
- один реалистичный fix или mitigation.

Не комментируй каждую строку кода. Лучше сделать код читаемым, а объяснение вынести в README.

## ADR

Используй ADR только для решений уровня репозитория: стратегия инструментов, CI snapshot model, dependency approach, cross-cutting conventions или решения, о которых будущий contributor разумно спросит "почему так?".

Не добавляй ADR для обычных bug examples.

Правила по ADR лежат в [AGENTS.md](AGENTS.md) и [.agents/skills/adr-writer/SKILL.md](.agents/skills/adr-writer/SKILL.md).
