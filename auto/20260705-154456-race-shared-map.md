# Race Shared Map Example

Commit: `e529e1e Add race shared map example`

## Context

The planned example was `race/shared_map`: shared metrics state accessed from multiple goroutines and detected by `go test -race`.

## Options

1. Use many goroutines writing directly to the same map.
   Score: 4/10. Very recognizable bug, but runtime `concurrent map writes` can crash before the race detector gives a useful report.

2. Use one writer and one reader on the map with `GOMAXPROCS=1`.
   Score: 5/10. Avoids the runtime crash, but the first attempt did not reliably overlap enough for `-race` to report.

3. Store mutable counter pointers in a map, initialize the key before concurrency, and race on `Counter.Value`.
   Score: 9/10. Keeps the example about shared map-owned state while avoiding runtime map internals and producing a stable race report.

4. Use a plain integer field instead of a map.
   Score: 6/10. Very stable, but loses the planned shared-map teaching point.

5. Normalize race snapshots with byte-wise `LC_ALL=C sed`.
   Score: 9/10. Handles race detector addresses that may contain bytes macOS sed rejects under UTF-8 locales.

## Chosen

Options 3 and 5.

The example uses `map[string]*Counter`; the map is initialized before concurrent access, then `Record` writes `Counter.Value` while `Snapshot` reads it.

## Why

This demonstrates the ownership mistake without letting the runtime map safety check dominate the lesson. The committed `race.logs` shows the conflicting write/read stacks from the race detector.
